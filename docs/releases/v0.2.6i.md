# v0.2.6i – Console Bookings Goes Live

## Purpose
Bookings has been sample data since the very first Console release. This makes it a real, live Firestore feature — the first half of the requested Bookings work (Console summary of all meetings). The second half — a Portal "My Meetings" view where customers see their own bookings and add notes — is a separate follow-up release, since it needs a Firestore rules change and new Portal UI.

## Changes
- Live `bookings` collection: Create, Edit, Delete, all working against Firestore, same pattern as Customers/Projects/Library.
- Real "New Booking" form: title, customer (locked once created, same reasoning as Projects), type, status, date, time, duration, internal notes.
- Date is a proper date field now (not free text), so bookings sort chronologically and display in a friendly format.
- Added (but not yet writable from the Portal) a `customerNotes` field, shown in the Console detail panel when present — laying the groundwork for the next release.
- Fixed two unrelated stale "Sample data" labels on the Dashboard (Projects, Library) that were wrong — both have been live for a while.

## Important: this is a manual log, not a Calendly sync
Creating, editing or deleting a booking here has **no effect on Calendly**, and a real Calendly booking does **not** appear here automatically. You keep booking sessions the same way as always; this just gives you a place to log what happened, for your own record and so it can eventually surface to the customer in the Portal. Automatic two-way sync (so Calendly bookings appear here without manual entry) would require adding a small backend piece (a Calendly webhook feeding a Cloud Function) — a bigger, separate infrastructure decision if you want it later.

## Firebase changes
None. Uses the existing `bookings` collection and existing admin-only Firestore rule (unchanged from before — this release doesn't touch Portal/customer access yet).

## Testing
1. Sign in as an administrator and open Bookings.
2. Confirm it now says "Loading bookings…" then shows a live (likely empty) list, not sample data.
3. Click "New Booking", fill in title, pick a customer, set a date/time, and save. Confirm it appears in the table.
4. Reload the page and confirm the booking is still there (proving it's real Firestore data, not in-memory only).
5. Open it and click "Edit booking". Confirm the customer field is locked, change something else, and save.
6. Click "Delete booking", confirm the prompt, and confirm it disappears.
7. Check the Dashboard's Bookings metric updates to match the live count.
8. Confirm the Dashboard's Projects and Library cards now say "Live data" instead of the old incorrect "Sample data".
