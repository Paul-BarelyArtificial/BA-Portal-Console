# v0.2.8b – Customer Upload Management

## Purpose
Console-side half of Customer Uploads (Portal v0.2.7): the tools to review, download and delete what customers submit, and the storage cost protection behind it.

## Changes
- Customer uploads land as `library` items with `source: "Customer"`, `status: "Draft"`, `visibility: "Internal"` — invisible to everyone until reviewed. Search "Customer" in the Library page to find pending submissions.
- Reviewing means using the existing Edit action to change status/visibility once you're happy with it — no separate "approve" button, it's the same Edit flow every other Library item already has.
- Downloading uses the existing file link, relabelled "Download file" for clarity.
- Deleting uses the existing "Delete permanently" action — now also refunds the uploading customer's quota automatically.
- Customer detail panel shows "Uploads used: X MB of 500 MB".

## Storage cost protection
- 20 MB per file, 500 MB total per customer.
- Enforced in the Firestore rules (the `library` record can't be created outside these bounds) **and** the Storage rules (the actual file write is rejected if it would go over) — genuine server-side enforcement, not just a client-side check a customer could bypass.
- A new `uploadStorageUsedBytes` field on each customer record tracks the running total. Customers can only increase it, by at most one file's worth per write — never decrease it, never touch any other field.

## Firebase changes — action required
Two files need to be published in Firebase Console before customer uploads will work:
1. **Firestore rules** — `docs/firestore.rules.txt` (updated: `library` create rule, `customers` update rule).
2. **Storage rules** — `docs/storage.rules.txt` (new file: adds the `customerUploads/{customerId}/...` path). This is the first time Storage rules have needed a Console-side change — your existing `library/` and `resources/` admin paths are untouched.

## Testing
1. Publish both rules files in Firebase Console.
2. As a test customer in the Portal, upload a document (see Portal v0.2.7 release notes for that half of testing).
3. In the Console, search "Customer" in the Library page and confirm the new item appears with status Draft, visibility Internal.
4. Open its detail panel and confirm you can download the file.
5. Edit it — change status to Published and visibility to whatever's appropriate — and confirm it now appears correctly in the Portal.
6. Open the uploading customer's detail panel and confirm "Uploads used" reflects the file size.
7. Delete the item and confirm the customer's "Uploads used" drops back down.
8. As the same test customer, try uploading a file over 20 MB and confirm it's rejected (client-side message, and if you check the browser's network/console you should see the Storage write itself was denied, not just the app's own check).
